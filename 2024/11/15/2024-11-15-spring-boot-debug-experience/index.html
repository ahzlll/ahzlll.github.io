Spring Boot 项目中的一次内存泄漏排查经历

发表于2024-11-15|更新于2025-11-10|Debug经验
|浏览量:

问题描述

刚开始学习Spring Boot不久，在做一个课程项目时遇到了一个奇怪的问题：应用运行一段时间后，内存占用越来越高，最终导致OOM（OutOfMemoryError）。作为一个后端开发新手，这是我第一次遇到内存泄漏问题，当时完全不知道从哪里入手排查。

排查过程

我完全不知道该怎么排查内存问题。在网上搜索后，了解到可以使用JVM工具来查看内存使用情况。

1. 初步分析

第一次使用 jmap 和 jstat 工具查看内存使用情况（之前完全不知道这些工具的存在）：

发现老年代（Old Generation）内存持续增长，Full GC频繁但效果不明显。

2. 生成堆转储

第一次听说还有这种工具！下载了 Eclipse MAT（Memory Analyzer Tool）来分析堆转储文件。刚开始用的时候完全看不懂，后来看了几篇教程才慢慢理解。

4. 根本原因

在Web应用中，线程池会复用线程。如果 ThreadLocal 的值没有在使用后清理，就会一直占用内存，导致内存泄漏。

方案二：使用 InheritableThreadLocal（不推荐）

对于线程池场景，InheritableThreadLocal 也不适用，因为线程是复用的。

经验总结

  1. ThreadLocal 必须清理：在使用完 ThreadLocal 后，一定要调用 remove() 方法
  2. 使用拦截器统一管理：在 Spring 中，可以通过拦截器统一清理 ThreadLocal
  3. 定期检查内存：生产环境要定期检查内存使用情况，及时发现问题
  4. 工具很重要：MAT、JProfiler 等工具是排查内存问题的利器

预防措施

  • 代码审查时重点关注 ThreadLocal 的使用
  • 添加内存监控告警
  • 定期进行压力测试，观察内存变化

教训：作为初学者，这次经历让我深刻认识到细节的重要性。一个小小的 ThreadLocal 清理问题，就能导致内存泄漏。写代码时一定要考虑资源的清理和释放，不能只关注功能实现。

这次debug经历让我学到了很多：

  • 学会了使用JVM工具排查问题
  • 理解了ThreadLocal的工作原理和注意事项
  • 知道了拦截器可以用来统一管理资源清理
  • 明白了监控和日志的重要性

虽然过程很痛苦，但收获很大。希望以后能少踩一些坑（虽然感觉不太可能 😅）！

第一次遇到内存问题的时候，真的是一头雾水，完全不知道从哪里下手。后来学会了使用JVM工具，才发现问题所在。
