Spring Boot 项目中的一次内存泄漏排查经历

发表于2024-11-15|更新于2025-11-10|Debug经验
|浏览量:

前言

这是我第一次在 Spring Boot 项目里认真面对“内存泄漏”这件事。最开始我只感觉应用越跑越慢，内存占用也一路上涨，最后直接 OOM（OutOfMemoryError）。当时我几乎不懂 JVM、GC，这篇文章就记录我作为一个新手，是怎么一路摸索、踩坑、再把问题解决的。过程中我也参考了很多社区文章和工具使用教程，把我自己的理解写下来，给以后的自己一个备忘。

我遇到的问题（现象）

  • 应用启动正常，跑一会儿后内存一直涨
  • Full GC 变得频繁，但内存就是降不下去
  • 压测或者多刷新几个接口，这个趋势更明显

那时候我只会看任务管理器里的内存曲线，完全不知道从哪里下手。后来我边搜边学，第一次用上了 jmap 和 jstat：

我看到老年代（Old Gen）在一直涨，Full GC 也救不回来，感觉问题不简单。

我是怎么排查的（一步步走）

1）生成堆转储

我把这个堆文件用 Eclipse MAT（Memory Analyzer Tool）打开。第一次用 MAT，页面信息很多，看得我一脸懵。后来根据一些教程，我主要看了 Dominator Tree 和 Leak Suspects。

2）MAT 给我的提示

MAT 里提示的一条线索是：有不少对象被 ThreadLocal 相关的引用链“挂住”了，释放不掉。顺着这个方向，我去翻了自己的代码，确实用了一个 ThreadLocal 存储每次请求的用户信息，但我从来没清过它。

4. 根本原因

Web 应用的业务线程来自“线程池”，线程会被复用。如果我只 set，不 remove，那么这个值就会“粘在这条线程上”。下次这条线程去处理别的请求时，之前的值还在，相关对象就可能一直被引用，久而久之就像漏水一样堆积起来了。

解决方案

方案一：在使用点用 try/finally 保证清理（直觉且有效）

方案二：在框架层统一兜底（我最终选了这个）

我后来更偏向用过滤器做“统一清理”，不管请求成功还是异常，finally 里都清一次，这样心里更稳。

（更靠近 Servlet 层的做法是基于 OncePerRequestFilter，它能用 try/finally 更直观地兜底清理；拦截器也能做到统一清理，按团队习惯选择即可。）

方案三：不要把 ThreadLocal 当“万能全局变量”

  • 能通过方法参数传的，就尽量别放 ThreadLocal
  • 异步任务、线程池切换时，不要指望 ThreadLocal 会“跟过去”
  • 日志里的 MDC 要记得配对清理（MDC.put() / MDC.clear()）

方案四：使用 InheritableThreadLocal？（我最后没有用）

对于线程池场景，InheritableThreadLocal 也不适用，因为线程是复用的。

经验总结

  1. ThreadLocal 必须清理：在使用完 ThreadLocal 后，一定要调用 remove() 方法
  2. 统一兜底很重要：用拦截器或过滤器在请求结束时清一次，心里更稳
  3. 遇事先量化：jstat 看 GC，jmap + MAT 看堆，别只凭感觉
  4. 少一点“隐式状态”：能参数传就参数传，别什么都塞 ThreadLocal
  5. 参考社区但要做验证：很多建议看起来正确，最好结合自己项目压测验证

预防措施

  • 代码审查时重点关注 ThreadLocal 的使用
  • 添加内存监控告警
  • 定期进行压力测试，观察内存变化

我给以后的自己（和同学们）的一份小清单

  • 有没有任何 ThreadLocal.set()，却没有相应的 remove()？
  • 异常分支、提前 return 会不会绕开清理？
  • 异步任务、线程切换有没有依赖 ThreadLocal？
  • 日志 MDC 有没有配对 clear？
  • 有没有一个全局的兜底清理（拦截器/过滤器）？

验证修复：用 ab / wrk 做一个小压测

我在修复后做了一个很简单的验证：用 ab & wrk 打一段时间请求，然后对比修复前后的 GC 与堆用量趋势。

  • 我观测的点

    • 修复后，Old Gen 不应持续单调上升；Full GC 次数不应异常增加
    • 压测结束后，堆占用能回落（配合 jstat 与一次堆 dump + MAT 复查）

  • 目标接口（示例）

    • 假设本地有一个 GET /demo，会在进入时 set ThreadLocal，并在 finally/过滤器中 remove

  • ab（Apache Bench）示例（Linux/Mac）

如果在修复后，ThreadLocal 相关的泄漏链条不再出现，且曲线表现正常，就说明这次修复基本有效。

尾声

这次的经历让我意识到：写功能只是第一层，资源的使用和清理同样重要。很多时候不是“我代码写错了”，而是“我忽略了运行时的那些细节”。我会继续把这些“踩坑—理解—解决”的经历记在这里，也感谢我在搜索路上读到的那些文章和讨论，让我能慢慢把这些知识串起来。希望以后能少踩一些坑（虽然感觉不太可能 😅）！
