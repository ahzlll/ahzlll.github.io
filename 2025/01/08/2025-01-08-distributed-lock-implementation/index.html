分布式锁的实现与思考

为什么需要分布式锁？

最近系统地看分布式系统资料时，我注意到一个高频问题：多个服务实例可能同时操作同一个资源。比如：

  • 库存扣减：多个订单同时扣减同一商品的库存
  • 优惠券发放：防止同一用户重复领取
  • 定时任务：防止多个实例同时执行

起初只是从资料里了解这些风险点感觉还是不太了解，所以我尝试做了小规模演练，验证分布式锁能有效规避这些并发冲突。

注意事项

  1. 锁的粒度：锁的粒度要合适，太粗会影响并发，太细会增加复杂度
  2. 超时时间：设置合理的超时时间，避免死锁
  3. 异常处理：确保在finally中释放锁，避免死锁
  4. 锁的value：使用唯一值（如UUID），确保只能释放自己加的锁
  5. Redis高可用：使用Redis集群或哨兵模式，保证可用性

总结

分布式锁是分布式系统中的重要组件。实现时要注意：

  • 原子性：加锁和设置过期时间要原子操作
  • 安全性：只能释放自己加的锁
  • 可重入性：同一线程可以重复获取锁
  • 自动续期：避免业务执行时间超过锁过期时间

对于生产环境，建议使用成熟的框架如 Redisson，它已经处理了各种边界情况。

思考：分布式锁看似简单，但实现起来有很多细节需要注意（真的很容易踩坑 😅）。在实际项目中，优先使用成熟的框架，避免重复造轮子。

第一次实现分布式锁的时候，想自己写一个，结果各种边界情况没考虑到，最后还是用了Redisson（真香！）。

和操作系统中的同步原语类比（PV、信号量、互斥锁等）

这部分是我在做分布式锁时的一个“恍然大悟”时刻。作为大学生，我第一反应就是把它和操作系统课上学到的 PV、信号量、互斥锁类比起来——这种跨课程联动的感觉挺新奇的，也帮助我更快地把概念落到工程实现上。

学操作系统时，我们最早接触并发控制往往从“临界区问题”出发，典型工具包括：

  • PV 原语（信号量）
    • P(wait) 与 V(signal) 操作分别对应对计数信号量的减/增；
    • 二值信号量可以充当互斥锁；计数信号量常用于资源池或限流；
    • 条件同步可通过“条件队列 + 信号量/条件变量”实现。（当时第一次敲 PV 代码，感觉把抽象“变成了按钮”。）
  • 互斥锁（mutex）与自旋锁（spinlock）
    • 自旋锁忙等，适合临界区极短、不可睡眠场景（如内核临界段）；
    • 互斥锁可睡眠阻塞，适合一般用户态临界区；
    • 可重入锁（recursive）允许同线程重复进入。（这点和 Redisson 的可重入锁概念直接对上了。）
  • 条件变量（condvar）
    • 与互斥锁配合，解决“等待某条件达成”的同步。
  • 经典问题与性质
    • 死锁四要素（互斥、占有且等待、不可剥夺、环路等待）；
    • 饥饿与公平性（队列、公平锁、优先级反转与优先级继承）。

把这些与分布式锁对照，有几点“相似点”和“根本差异”很有帮助：

相似点

  • 都是为了解决“互斥进入临界区”“顺序化访问共享资源”；
  • 都强调正确释放、超时与异常场景的处理；
  • 都关心公平性、可重入性、死锁/饥饿等经典问题。

根本差异（分布式语境下的新增复杂度）

  • 失败模型不同：
    • 本地锁默认“同一进程、同一内存空间”，失败大多是线程崩溃；
    • 分布式锁要面对“进程崩溃 + 节点宕机 + 网络分区/抖动 + 时钟漂移”。（第一次遇到网络分区这个词时我是真愣住了。）
  • 可见性与一致性边界不同：
    • 本地锁由 CPU/内存一致性协定保障；
    • 分布式锁的“可见性”由远程存储或共识系统保障（Redis、ZK、Etcd、数据库等），涉及复制、故障切换与一致性级别。
  • 时间与租约（lease）：
    • 操作系统里很少用“租约时间”确保锁最终释放；
    • 分布式锁必须假设持有者可能失联，只能用 TTL/租约 + 续期（看门狗）保证“最终可回收”。
  • 所有权与幂等：
    • 本地释放基本由“线程身份”判定；
    • 分布式必须用“唯一 value/令牌”来校验所有权，避免误删他人锁；
    • 重试需设计为幂等，避免网络重放导致的重复执行。
  • 顺序与公平：
    • 本地可用 FIFO 队列、公平锁；
    • 分布式要考虑跨节点排队与时钟无序，公平性成本更高，通常退而求其次（尽量减少冲突、指数退避）。我一开始天真地以为“加个公平锁”就行，后来发现完全不是一个数量级的问题。
  • 死锁处理：
    • 本地可借助可剥夺策略、锁顺序、超时；
    • 分布式更依赖“短租约 + 超时 + 监控告警 + 幂等补偿”，跨系统的锁顺序约定更难落实。

从 OS 原语得到的设计启发

  • 用“二值信号量 ≈ 互斥锁”的思路理解 Redis 分布式锁最直观，但要补齐网络与租约带来的不确定性；
  • 条件变量的“等待-通知”在分布式里往往转化为事件/消息（但不要把锁当消息队列）；
  • 优先级反转在分布式中可类比为“关键任务拿不到锁被长事务占用”，工程解法更偏向于缩短临界区、拆分任务与限流；
  • 避免死锁的“固定加锁顺序”在多服务/多资源的分布式场景尤为重要（如先用户后订单，所有服务遵循同一顺序）。

结合本文实现的具体落点

  • 使用唯一 value（如 UUID）绑定持有者，释放前校验，类比“锁所有权”；
  • 通过 Lua 保证“检查 + 删除”的原子性，类比本地“临界区不可分割”；
  • 看门狗续期是“分布式租约”的核心配套，弥补网络与执行时间不确定；
  • 控制临界区大小与耗时，减少自旋/冲突概率；
  • 对业务侧提供幂等与重试策略，兜底网络异常；
  • 若需要强公平/强一致，可考虑 ZooKeeper/Etcd 排队式锁或基于共识的方案，取舍吞吐与复杂度。

用我自己的话总结一下：PV/互斥锁教会我们“如何正确进出临界区”，而分布式锁是把这套逻辑搬到“跨进程、跨机器、可能会失联”的世界里——于是就不得不引入租约/续期、所有权校验、幂等这些额外“护栏”。当我把这两块拼在一起时，许多看似“玄学”的线上问题就有了可解释的因果链，成就感还挺强的。
