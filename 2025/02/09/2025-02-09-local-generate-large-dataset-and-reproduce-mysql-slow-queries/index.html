本地快速造 100 万+ 数据并复现 MySQL 慢查询

发表于2025-02-09|更新于2025-11-10|Debug经验
|浏览量:

目标与适用场景

这篇记录我在本地快速造出 100 万+ 行数据，并稳定复现“慢查询”的做法。适用于：

  • 练习慢查询日志定位与 EXPLAIN 解读
  • 评估索引设计对查询性能的影响
  • 做作业/写报告/准备面试的本地实验

不追求“绝对真实业务数据”（其实是没办法），而是“足够规模 + 分布可控 + 可重复”。

环境与基础设置

  • MySQL 8.0（其他版本思路一致）
  • 建议开启慢查询日志（练习时阈值调小更易抓到慢 SQL）

Windows 常见日志路径示例：

  • 官方 MSI：C:\ProgramData\MySQL\MySQL Server 8.0\Data\slow.log
  • XAMPP：C:\xampp\mysql\data\slow.log
  • WAMP：C:\wamp64\bin\mysql\mysql8.0.xx\data\slow.log

精确定位路径：

若 slow_query_log_file 为相对路径，则在 datadir 下；若无法写入，请调整目录权限或改到可写路径。

基础表结构（示例）

说明：这里先放一些必要索引，后续会通过是否添加“合适的复合索引”来复现慢与快的差异。

造数方法一：递归 CTE（MySQL 8.0+）

优点：无需外部工具，SQL 一把梭；可控数量。
缺点：插入速度一般，不建议一次性拉很大批次，可分段执行。

小提示：可按 10 万为单位分批插入，降低单次事务开销。

造数方法二：数字表/笛卡尔积

优点：快、可控、纯 SQL；
缺点：写法略“硬核”，但非常实用。

造数方法三：脚本生成 CSV + LOAD DATA

优点：速度快、占用低；可用任意语言生成更拟真数据分布。
缺点：需要磁盘空间，需开启本地导入权限。

客户端与内存使用注意

  • MySQL 不会把 100 万行一次性装进内存，主要瓶颈在磁盘与是否命中索引
  • 客户端避免一次性取回巨量结果：
    • CLI：mysql --quick
    • 程序：使用服务端游标/流式读取（JDBC fetchSize、Node.js 流、Go rows.Next）

常见问题

  • 日志文件没有生成：检查 slow_query_log=ON、log_output=FILE、slow_query_log_file 路径可写
  • LOAD DATA 被拒绝：需要 --local-infile=1 或授予 FILE 权限；Windows 路径用双反斜杠
  • 插入很慢：分批提交、暂时关闭外键/唯一性检查、考虑先导入后建索引
  • 磁盘占用超预期：数据+二级索引会放大；评估字段长度与索引数量
