Spring事务隔离级别的一次踩坑经历

发表于2025-03-20|更新于2025-11-09|Debug经验
|浏览量:

问题描述

在开发一个订单系统时，遇到了一个奇怪的问题：

  1. 用户下单后，库存扣减成功
  2. 但订单状态没有更新为”已支付”
  3. 查看日志，没有发现任何异常

代码场景

问题分析

1. 事务隔离级别

查看数据库配置，发现使用的是READ COMMITTED隔离级别。

2. 并发场景

假设有两个请求同时处理：

时间线：

  1. T1: 事务A开始，查询库存为100
  2. T2: 事务B开始，查询库存为100
  3. T3: 事务A扣减10，库存变为90，但未提交
  4. T4: 事务B扣减10，库存变为90（基于100计算），但未提交
  5. T5: 事务A提交，库存为90
  6. T6: 事务B提交，库存为90（应该是80！）

问题：出现了超卖！

3. 隔离级别对比

隔离级别 脏读 不可重复读 幻读 说明
READ UNCOMMITTED 可能 可能 可能 最低级别
READ COMMITTED 不可能 可能 可能 MySQL默认
REPEATABLE READ 不可能 不可能 可能 可重复读
SERIALIZABLE 不可能 不可能 不可能 最高级别，性能最差

解决方案

方案一：使用REPEATABLE READ（不推荐）

问题：虽然能解决不可重复读，但性能较差，且可能出现死锁。

方案二：使用悲观锁（SELECT FOR UPDATE）

SQL：

优点：简单直接，保证数据一致性
缺点：性能较差，可能造成锁等待

方案三：使用乐观锁（推荐）

SQL：

优点：性能好，适合高并发场景
缺点：需要重试机制，实现稍复杂

方案四：使用分布式锁

最终方案

结合业务场景，我们采用了乐观锁 + 分布式锁的组合方案：

  1. 分布式锁：防止并发请求同时进入
  2. 乐观锁：数据库层面保证数据一致性

经验总结

  1. 理解隔离级别：不同隔离级别解决的问题不同，要根据业务选择
  2. 并发控制：高并发场景下，要考虑并发控制机制
  3. 锁的选择：
    • 悲观锁：适合写多读少的场景
    • 乐观锁：适合读多写少的场景
    • 分布式锁：适合分布式环境
  4. 性能平衡：在数据一致性和性能之间找到平衡

教训：事务隔离级别不是万能的，高并发场景下需要结合锁机制来保证数据一致性。理解各种锁的适用场景很重要。
